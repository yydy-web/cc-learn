## 场景概述

产品 VP 周五下午丢来一句话："下周三要给 CEO 汇报，我们需要一个文档协作工具的选型分析——Notion、Confluence、飞书文档，哪个更适合我们的团队？"

传统的做法：打开三个产品的官网、找评测文章、看用户评价、手动填对比表、分析优劣势。竞品调研类工作有个特点——工作量线性增长（每多一个竞品就多一倍的页面要翻），但分析深度和覆盖面成反比（看得越多，每个竞品看得越浅）。3 个竞品 × 每个竞品至少翻 10 篇文章 + 5 个用户评价 = 至少 1-2 天的工作量，最后产出一份"看起来全面但细节经不起推敲"的对比表。

用 Claude Code Agent 的并行调研模式：**3 个 Advocate Agent 各负责一个竞品，深度调研功能矩阵；1 个 Judge Agent 按你预设的维度和权重综合打分**。整个过程 5-10 分钟，产出的每个数据点都有来源链接可追溯。

## 为什么用多 Agent 并行调研

竞品调研不是"搜一下竞品有什么功能"——是要在多个维度上做**结构化对比**。一个人同时调研 3 个竞品，脑子在三个产品之间来回切换，不仅慢，而且容易把 A 的功能记成 B 的。

| 对比维度   | 人工调研                           | 多 Agent 并行调研                   |
| ---------- | ---------------------------------- | ----------------------------------- |
| 耗时       | 4-8 小时（搜 + 读 + 填表 + 分析）  | 5-10 分钟（定义维度 + 审阅结论）    |
| 深度一致性 | 第一个竞品看得最细，第三个开始疲劳 | 每个 Agent 只盯一个竞品，深度一致   |
| 来源可追溯 | 容易忘记记链接                     | 每个数据点附来源 URL                |
| 偏见控制   | 个人偏好影响大                     | 3 个 Advocate + 1 个 Judge 对冲偏见 |
| 复用性     | 一次性产出                         | 矩阵模板可复用，下次换竞品只改名字  |

核心不是"AI 替你调研"，而是**多视角并行覆盖 + 结构化输出**。你只需要定义"比什么"和"怎么打分"，剩下的搜索、阅读、整理工作全部并行处理。

## 前置准备

- **候选产品明确**：3-4 个竞品名称和官网 URL。候选太多会导致调研发散，太少起不到对比效果
- **评估维度已定**：建议 4-6 个维度，包含功能覆盖、易用性、集成生态、定价、安全合规、用户口碑等
- **权重已商定**：这是最关键的一步——权重决定了最终打分的倾向性，必须在使用方内部达成共识后再启动 Agent，不能让 AI 自己构造维度权重

:::tip
功能对比矩阵的粒度建议控制在一级功能（如"实时协作""权限管理""API 开放程度"），不要细到二级功能（如"能否设置行级权限"）。太细会导致调研时间暴涨而增量价值递减。
:::

## 完整交互过程

### Step 1：定义调研维度和输出模板

先把调研标准说清楚。这是整个流程的质量控制点——维度越具体，Agent 输出越聚焦。

```text
对以下 3 个文档协作工具做竞品功能矩阵分析：

**候选产品**
1. Notion（notion.so）
2. Confluence（atlassian.com/confluence）
3. 飞书文档（feishu.cn/product/docs）

**调研维度**
1. 核心功能覆盖 — 文档编辑、表格、数据库、看板、日历视图
2. 协作能力 — 实时协同编辑、评论与批注、版本历史、@提醒
3. 权限与安全 — 页面级/空间级权限、SSO、审计日志、数据加密
4. 集成生态 — 第三方集成数量、API 开放度、自动化（webhook/zapier）
5. 定价与性价比 — 免费版限制、团队版价格、企业版起售价
6. 用户口碑 — 从 G2/Capterra/知乎 各提取 5 条典型评价

**输出要求**
- 每个维度输出一个对比表格（竞品 × 指标），附简要分析（2-3 句）
- 表格后附信息来源（文章标题 + URL）
- 调研范围为近 6 个月内的信息，标注"信息时效截止日期"
```

### Step 2：3 个 Advocate Agent 并行调研

三个 Agent 同时启动，各自认领一个竞品，按统一维度进行深度调研。以下是 Advocated Agent 的输出摘要。

**Agent A — Notion 调研报告**

```text
## Notion 功能矩阵调研

调研日期：2026-07-06 | 信息来源：官网 + G2(30+条评价) + 知乎(15+条讨论)

### 1. 核心功能覆盖
Notion 的核心差异在于"All-in-one"——同一页面内可嵌入文档、数据库、看板、
日历、时间线，彼此之间通过 database relation 关联。但这也意味着：
- 单个页面的加载速度受嵌入块数量影响，超过 20 个 block 后明显变慢
- 数据库计算字段能力弱于 Airtable，不支持公式引用其他数据库

### 2. 协作能力
- 实时协同编辑：支持，但超过 5 人同时编辑时延迟感知明显（G2 多条反馈）
- 评论：支持行内评论，但不支持评论状态追踪（已解决/未解决）
- 版本历史：免费版保留 7 天，Team 版保留 30 天

### 3. 权限与安全
- 页面级权限：支持（公开/仅成员/仅邀请），但空间级权限与页面级权限的
  交互逻辑复杂，用户反馈"权限继承看不太懂"（G2 高频提及）
- SSO：仅 Enterprise 版支持 SAML/SCIM

### 4. 集成生态
- 第三方集成：70+ 个（Slack、GitHub、Figma、Jira 等）
- API：REST API 全功能覆盖，官方 SDK 覆盖 JS/Python
- 自动化：内置 Database Automations（仅 Enterprise），第三方用 Zapier/Make

### 5. 定价
| 版本     | 价格（每人/月） | 关键限制 |
| -------- | --------------- | -------- |
| Free     | $0              | 7 天版本历史、5MB 文件上传 |
| Plus     | $10             | 30 天版本历史、无限上传 |
| Business | $15             | 高级权限、私有空间 |
| Enterprise | 议价           | SAML/SCIM、审计日志、专属成功经理 |

### 6. 用户口碑摘要
- 👍 "最适合个人知识管理和小团队协作，灵活度吊打所有竞品" — G2
- 👍 "数据库 + 多视图切换是杀手级功能，取代了我们的 Excel + Trello" — 知乎
- 👎 "超过 10 人团队后加载变慢，搜索也不如 Confluence 精准" — G2
- 👎 "缺少文档审批流，不能替代正式的文档管理系统" — 知乎

来源：
- notion.so/pricing — 官网定价页
- g2.com/products/notion/reviews — G2 用户评价
- zhihu.com/question/notion-vs-confluence — 知乎讨论
```

**Agent B — Confluence 调研报告** 和 **Agent C — 飞书文档调研报告** 的输出格式同上，分别按 6 个维度覆盖各自竞品的信息。

### Step 3：Judge Agent 按维度综合打分

三个调研报告出来后，Judge Agent 不偏向任何一方，按预设的 6 个维度和权重打分。

```text
## 综合评分报告（5 分制）

### 评分矩阵

| 维度 (权重)            | Notion | Confluence | 飞书文档 |
| ---------------------- | ------ | ---------- | -------- |
| 核心功能覆盖 (25%)      | 4.5    | 3.5        | 3.5      |
| 协作能力 (25%)          | 3.5    | 4.0        | 4.5      |
| 权限与安全 (15%)        | 3.0    | 5.0        | 3.5      |
| 集成生态 (15%)          | 3.5    | 5.0        | 3.0      |
| 定价与性价比 (10%)      | 4.0    | 3.0        | 4.0      |
| 用户口碑 (10%)          | 4.0    | 3.5        | 4.0      |
| **加权总分**            | **3.775** | **4.025** | **3.775** |

### 差异化分析

**Confluence 的优势在企业级**：
- 权限和审计能力是三个中最强的——这恰恰是大团队最在乎的
- Jira 生态绑定使它在"研发团队"场景中不可替代
- 但定价偏高，且"重"——小团队会觉得功能冗余

**飞书文档的优势在实时协作**：
- 多人实时编辑的流畅度是三个中最好的，背后是字节跳动的底层协同引擎
- 和飞书 IM/日历/审批的原生集成是降维打击——Notion 和 Confluence 做不到
- 劣势是国际化程度不够，海外团队使用受限

**Notion 的优势在灵活性**：
- All-in-one 的设计理念使其能覆盖最多场景——个人笔记、团队 wiki、项目管理
- 社区模板生态丰富，上手门槛低
- 劣势是企业级能力偏弱，权限模型不够严谨

### 推荐结论

**团队规模决定选型**：
- 50 人以下、追求灵活 → Notion（性价比最高）
- 50-200 人、已有 Atlassian 生态 → Confluence（零迁移成本）
- 50-200 人、使用飞书办公 → 飞书文档（原生集成是核心价值）
- 200 人以上、强合规需求 → Confluence（权限审计无可替代）
```

### Step 4：团队基于报告做决策

最终团队选择飞书文档。决策理由：

- 团队已在飞书办公（IM、日历、审批），加飞书文档是"顺带"而非"新增一个工具"
- 协作流畅度对日常使用的影响权重被低估了——不流畅的编辑器会让员工潜意识抗拒写文档
- Notion 的"灵活"在本场景中是劣势：太灵活意味着每个人建的页面结构都不一样，长期维护成本高

决策记录（评分表 + 选择理由 + 切换成本评估）存档到 `docs/decisions/`。

## 要点总结

1. **调研维度必须在启动 Agent 前确认**。维度是质量控制的核心杠杆——模糊的维度产出模糊的结论。建议控制在 4-6 个维度，兼顾覆盖面和 Agent 的处理深度。
2. **每个数据点要求附来源链接**。Agent 的调研结论不可盲信——附来源让你可以在 30 秒内验证任何一个数据点。没有来源的数据等于听说的。
3. **Judge Agent 的权重不能事后改**。如果你对评分结果不满意然后调整权重，等于在"让数据迁就结论"。权重必须在启动 Judge 之前拍板，不满意结论应该重新审视维度和候选产品池。
4. **竞品分析结果存档为团队知识**。这份功能矩阵不仅用于这次选型——3 个月后新人问"我们为什么不用 Notion"，你不需要再解释一遍。
5. **"信息时效截止日期"很重要**。竞品的功能和定价在快速迭代中，调研报告中标注"信息截至 X 月 X 日"是一项容易被忽略但关键的专业习惯。

## 变体与延伸

- **投资赛道分析**：把竞品替换为目标赛道的公司（如 AI Agent 赛道 5 家创业公司），维度改为团队背景/融资阶段/技术壁垒/客户数量/商业模式，输出投资 memo
- **供应商选型**：替换为 SaaS 供应商（如邮件服务 SendGrid vs Resend vs Postmark），维度改为送达率/API 设计/定价/SLA/合规认证
- **技术标准选型**：替换为技术标准（如 WebSocket vs SSE vs WebTransport vs gRPC stream），维度改为浏览器兼容性/服务端复杂度/断线重连/吞吐量
- **合作伙伴评估**：替换为潜在合作伙伴，维度改为市场覆盖/技术互补性/合作模式/排他风险/过往案例

每次只需改三个参数：候选列表、评估维度、权重。流程完全复用。
